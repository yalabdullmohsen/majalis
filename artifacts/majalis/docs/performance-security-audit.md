# تدقيق الأداء والأمان (2026-07-16)

منهجية هذا التدقيق: كل نتيجة أمنية هنا **تحقَّقنا منها فعليًا ضد قاعدة
بيانات الإنتاج الحية** (عبر `npx supabase db query --linked`، بما في ذلك
اختبار صلاحيات حقيقية بـ `BEGIN; SET LOCAL ROLE anon; ...; ROLLBACK;`)، لا
من قراءة ملفات SQL المُرحَّلة (migrations) في المستودع فقط. هذا فرّق النتيجة
جوهريًا: عدة "ثغرات" ظهرت من قراءة الملفات الثابتة تبيَّن أنها **مُصلَحة
فعليًا في الإنتاج** (عبر hotfix سابق أو migration لاحقة لم تنعكس بوضوح في
اسم الدالة/السياسة بالملف الأصلي) بمجرد التحقق المباشر. الدرس: **لا تثق
بملفات migration المُرحَّلة وحدها لتقييم أمان مشروع مجالس — تحقّق دائمًا من
الحالة الحية.**

## الأداء

### حجم الحزمة الأولى (initial bundle) — لم يُصلَح، موثَّق فقط

القياس الفعلي من `pnpm run build` (Vite، gzip):

| المورد | الحجم (gzip) |
|---|---|
| `index-*.js` (المدخل الرئيسي) | 163.1 KB |
| `vendor-*.js` (مُحمَّل مسبقًا eager) | 78.3 KB |
| `supabase-*.js` (مُحمَّل مسبقًا eager) | 54.6 KB |
| `icons-*.js` (مُحمَّل مسبقًا eager) | 21.7 KB |
| `query-*.js` (مُحمَّل مسبقًا eager) | 10.1 KB |
| **إجمالي JS الأولي** | **~328 KB** |
| `index-*.css` (ملف CSS واحد شامل) | **~246 KB** |
| **الإجمالي الفعلي (JS+CSS) قبل أي محتوى** | **~574 KB gzip** |

هذا يتجاوز هدف "≤300KB gzip للحزمة الأولية" المُستهدَف بنحو الضِّعف عند
احتساب CSS معه. السبب الجذري: ملف CSS واحد غير مُقسَّم (`index.css` يستورد
أربع طبقات تراكمية — `design-system.css` + `elite-2026.css` (~35 ألف سطر،
المرجعية الفعلية) + `majalis-v2.css` + `modern-2026.css`، إجمالًا ~52 ألف
سطر) يُحمَّل بالكامل على كل صفحة بصرف النظر عن استخدامها الفعلي.

**لماذا لم يُصلَح في هذه الجلسة**: تقسيم/تنقية هذا الملف بأمان يتطلب تتبّع
أي كلاس يُستخدَم في أي صفحة من أصل 164 صفحة، ثم تحقّقًا بصريًا (Playwright
screenshot) لكل صفحة متأثرة — نفس المنهجية المُستخدَمة سابقًا هذه الجلسة
لإصلاح 8 أعطال CSS فردية، لكن مُطبَّقة الآن على النطاق الكامل بدل حالات
معزولة. المحاولة العشوائية (blind refactor) على ملف بهذا الحجم كانت لتحمل
مخاطرة كسر بصري حقيقية عبر عشرات الصفحات بلا قدرة على فحصها كلها في هذه
الجلسة. **هذا العنصر يبقى بندًا موثَّقًا وواضحًا لعمل مستقبلي مخصَّص**، لا
تخمينًا مُموَّهًا بإصلاح جزئي.

**توصية للعمل المستقبلي**: تقسيم `elite-2026.css` (الملف المهيمن) حسب
الصفحة/الميزة عبر Vite's CSS code-splitting (استيراد CSS داخل كل مكوّن
View بدل استيراد عام في `index.css`)، مع فحص Playwright شامل (لقطات لكل
مسار رئيسي، فاتح/داكن) قبل وبعد لكل دفعة تقسيم.

### npm audit — غير قابل للتشغيل حاليًا (قيد خارجي)

كل من `pnpm audit` و`npm audit` فشلا: نقطة نهاية فحص الثغرات القديمة في
سجل npm (`registry.npmjs.org/-/npm/v1/security/audits`) أُوقفت (HTTP 410،
"retired, use bulk advisory endpoint")، و`npm audit` يتطلب `package-lock.json`
غير موجود في مشروع يستخدم pnpm (`pnpm-lock.yaml`). هذا قيد خارجي في أدوات
سلسلة التوريد، لا في المشروع نفسه — يحتاج أداة بديلة (مثل `pnpm audit`
بعد تحديث pnpm لدعم نقطة النهاية الجديدة، أو `osv-scanner`) لفحص الثغرات
المعروفة في الاعتماديات مستقبلًا.

## الأمان — نتائج التحقق المباشر من الإنتاج

### ✅ مُتحقَّق أنها آمنة فعليًا (رغم أن الملفات الثابتة أوحت بخلاف ذلك)

- **`governance_user_roles`**: السياسات الحية `governance_roles_admin_all`
  (`USING/WITH CHECK: is_admin()`, `authenticated`) و`governance_roles_self_read`
  (`user_id = auth.uid()`, SELECT فقط) — لا مسار تصعيد صلاحيات. ملف
  `hotfix_rls_privilege_escalation_2026-07-13.sql` إما طُبِّق أو استُبدِل
  لاحقًا بسياسات أنظف — النتيجة الحية آمنة بصرف النظر عن التاريخ الدقيق.
- **`user_submissions`**: `user_sees_own` الحية هي `auth.uid() = user_id`
  فقط — بلا "OR auth.uid() IS NULL" الخاطئة المذكورة في ملف الـ hotfix.
- **مُشغِّل `profiles` (`protect_bootstrap_owners`)**: مُتحقَّق سابقًا (جلسة
  2026-07-15، راجع الذاكرة) عبر `BEGIN…ROLLBACK` حي أنه يرفض التصعيد في
  كلا الاتجاهين. لا زال ساري المفعول.
- **`open_api_keys`**: RLS مُفعَّلة، صفر سياسات = رفض كامل افتراضيًا لأي
  دور غير service_role. آمن (الحد الأقصى من التقييد).
- **`admin_audit_logs`**: RLS مُفعَّلة فعليًا في الإنتاج (خلافًا لما أوحى به
  عدم وجود `ENABLE ROW LEVEL SECURITY` في أي ملف migration مُرحَّل) بسياستين:
  `admin_read_audit`/`admin_insert_audit` كلتاهما مقيَّدة بـ `is_admin()`.
- **`content_views`**: سياسة SELECT الصفّية `USING(true)` تبدو مقلقة للوهلة
  الأولى (تحوي عمود `user_id` حساس)، **لكن** GRANT على مستوى العمود يمنع
  فعليًا قراءة `user_id` لأي دور غير service_role — تحقَّق تجريبيًا:
  `SET LOCAL ROLE anon; SELECT user_id FROM content_views` ← `permission
  denied for table content_views`. بينما `SELECT content_id, content_type,
  viewed_at` وَ`count(*)` يعملان (مطلوبان فعليًا لميزتي "عدد المشاهدات"
  العامة ولوحة تحكم المشرف). هذا تصميم مقصود وسليم بالفعل — لم يُغيَّر.
- **`qa_category_corrections`**: الجدول غير موجود أصلًا في قاعدة بيانات
  الإنتاج — لا شيء يحتاج حماية.

### 🔧 أُصلِح فعليًا في هذه الجلسة (كود، لا قاعدة بيانات)

- **`/api/submissions` و`/api/client-error-log` كانا بلا أي حد لمعدّل
  الطلبات** رغم كونهما نقطتَي كتابة عامتين بلا مصادقة (أكثر المسارات عرضة
  لإساءة الاستخدام — إغراق قائمة المراجعة أو سجل الأخطاء). أُضيف
  `createRateLimiter` بحدود 8 طلبات/دقيقة (submissions) و15 طلبًا/دقيقة
  (client-error-log) في `lib/api-dispatch.mjs`، بنفس نمط المسارات المحمية
  الأخرى (`/api/citations`, `/api/rag`, إلخ).

### 📋 موثَّق كقيد/فجوة (لم يُغيَّر — يحتاج قرارًا أو نطاقًا أوسع)

- **لا طبقة تحقّق مُوحَّدة (Zod أو مكافئ) في `lib/api-handlers/`** — كل
  معالج يتحقق يدويًا (بعضها جيد كـ `submissions.js`، بعضها سطحي). إضافة
  Zod شاملة عبر عشرات المعالجات نطاق أوسع من إصلاح نقطي، ويحتاج مراجعة كل
  معالج على حدة لتفادي كسر شكل طلبات صحيحة حاليًا — لم يُنفَّذ في هذه
  الجلسة، موصى به كمهمة منفصلة.
- **جداول القياس العامة (`content_views` وما شابه) بلا تحقّق شكل صارم على
  مستوى RLS** لعمليات INSERT العامة (`WITH CHECK: true`) — مقبول تصميميًا
  لجداول تتبّع/تحليلات لا تحوي محتوى حسّاسًا بالقراءة العامة (بعد التأكد من
  حماية الأعمدة الحساسة كما أعلاه)، لكن بلا حد لمعدّل الطلبات على مستوى
  PostgREST مباشرة (خارج نطاق `lib/api-dispatch.mjs` كليًا لأنها لا تمر
  بخادم التطبيق، بل PostgREST مباشرة من العميل) — قيد بنيوي في تصميم
  Supabase-direct-from-client لا حل سريع له.
- `error_reports` و`search_queries` (بالاسمين الدقيقين المستخدَمين في
  `src/lib/supabase.ts`) **غير موجودين فعليًا** في قاعدة بيانات الإنتاج
  (تأكَّد بالاستعلام المباشر) — على الأرجح استعلامات لوحة تحكم المشرف
  تفشل بصمت لهذين الحقلين تحديدًا وتُرجِعان بيانات فارغة. هذا عطل وظيفي
  في لوحة التحكم لا ثغرة أمنية (لا بيانات موجودة لتُسرَّب) — خارج نطاق
  هذا التدقيق الأمني، يُترَك كملاحظة لمراجعة لوحة تحكم المشرف مستقبلًا.
