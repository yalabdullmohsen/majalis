## 2) بنية البيانات الحالية — **لا Drizzle ولا Postgres للنص القرآني**

خلافًا لافتراض البرومبت ("Supabase + Drizzle ORM")، **لا يوجد أي جدول `quran_ayahs`/`quran_surahs`/`quran_pages` في قاعدة البيانات إطلاقًا**. النص القرآني بالكامل يُخدَّم من **ملفات JSON ثابتة مبنية مسبقًا** في `public/data/quran/`:

- `surah-001.json` … `surah-114.json` (114 ملفًا، نص Uthmani فقط — **لا QPC V2 glyph text**، لا بيانات كلمة-بكلمة، لا `line_number`)
- `pages-manifest.json` (604 صفحة: أي سور/نطاقات آيات تقع في كل صفحة — **مستوى السورة/الآية فقط، لا مستوى السطر/الكلمة**)
- `page-juz-index.json`, `quran-position-index.json`, `mutashabihat-index.json`, `manifest.json`
- الحجم الإجمالي: **6.3 ميجابايت فقط** لكل بيانات القرآن الحالية — يؤكد غياب أي بيانات على مستوى الكلمة/السطر.

**الخلاصة:** التخطيط الحالي (`MushafPage.tsx`) يعرض نصًا مستمرًا مُبرَّرًا (justified paragraph) بخط Amiri Quran واحد لكل الصفحات — **وليس** إعادة إنتاج دقيقة لأسطر مصحف المدينة الخمسة عشر. هذا هو الفارق الجوهري الذي يستهدفه هذا البرومبت.

## 3) ما سيبقى دون تغيير (لا حذف)

- `stripEmbeddedBismillah`, `fetchSurahDetail`, `getSurahMeta` في `quran-api.ts` — تُستخدَم من مكوّنات أخرى كثيرة (`ExploreAyahPanel`, `RecitationTestPage`, البحث الموحّد) لا علاقة لها مباشرة بعرض الصفحة.
- `AyahActionSheet.tsx` — يُعاد استخدامه/تكييفه، لا يُعاد بناؤه من الصفر.
- مسارات `/quran/*` الأخرى (radio, memorization, tajweed, surah-stories, revelation-order) — خارج نطاق هذا البرومبت تمامًا.
- **جداول Supabase الحالية للتسميع بالذكاء الاصطناعي** (`quran_recitation_ai_test_v1/v2`، جداول `recitation_sessions`/`recitation_errors`/`recitation_settings`) — انظر البند الحرج أدناه.

## 4) ⚠️ اكتشاف حرج: ميزة "التسميع بالذكاء الاصطناعي" **موجودة وتعمل فعليًا حاليًا** — ليست فارغة

البرومبت (القسم 9) يطلب بناء **Placeholder معماري فقط** خلف Feature Flag معطّل، بافتراض أن الميزة غير موجودة. **هذا الافتراض غير صحيح اعتبارًا من هذه اللحظة**:

- المسار `/quran/recitation-test-ai` **منشور فعليًا على production** (`www.majlisilm.com`)، مع مزوّد تعرّف صوتي خادمي حقيقي (Groq `whisper-large-v3` عبر `/api/recitation-transcribe`) يعمل على كل من الويب وتطبيق iOS الأصلي، بالإضافة لمزوّد Web Speech API على المتصفح ومزوّد جهازي على iOS الأصلي.
- الجداول موجودة وتعمل: `recitation_sessions`, `recitation_errors`, `recitation_settings` (هجرات `quran_recitation_ai_test_v1.sql` و`v2_constraint_widening.sql`).
- الميزة تشمل: أوضاع تسميع متعددة (كامل/بمساعدة/كلمة بكلمة/مصحف تفاعلي/اختبار معلّم/حر)، مستويات تنبيه (لطيف/متوسط/فوري/معلّم حقيقي)، نظام ثقة ثلاثي، شارات إنجاز، شاشة خصوصية، معالجة مقاطعات صوتية حقيقية.

**القرار المتَّخذ لهذا الجرد:** تنفيذ القسم 9 حرفيًا (تفريغ الميزة إلى "قريبًا" خلف علم معطّل) يُعَدّ **تراجعًا حقيقيًا عن ميزة قائمة تعمل** — يخالف القاعدة غير القابلة للتفاوض رقم 3 ("لا حذف لأي ميزة قائمة"). **سيُتخطى القسم 9 من التنفيذ**، وتبقى الميزة الحالية كما هي، مع التأكد فقط أن زر "🎙️ التسميع" في تصميم المصحف الجديد يستمر بالربط الصحيح بـ`/quran/recitation-test-ai` (كما يفعل الحالي عبر "سمّع" CTA) بدل تجميد الميزة.

## 5) خريطة المسارات: قبل → بعد

| قبل | بعد | ملاحظة |
|---|---|---|
| `/mushaf` | `/mushaf` (نفسه) | تخطيط داخلي جديد كليًا، لا تغيير مسار |
| `/mushaf/:surah` | `/mushaf/:surah` (نفسه) | يُحوَّل لأول صفحة السورة كما الآن |
| — | `/mushaf/page/:n` (جديد) | Deep link مباشر لصفحة برقمها |
| — | `/mushaf/ayah/:key` (جديد) | Deep link مباشر لآية بمفتاحها |
| — | `/mushaf/tasmee` | **متروك** — الميزة الحقيقية موجودة أصلًا على `/quran/recitation-test-ai`، لا حاجة لمسار مكرَّر فارغ |

## 6) نسخة احتياطية قبل الترحيل

نظرًا لعدم وجود جداول DB يُرحَّل منها (البيانات ملفات JSON ثابتة فقط)، النسخة الاحتياطية = **الإبقاء على `public/data/quran/*.json` الحالية دون حذف** (تبقى كمصدر احتياطي/fallback وتُستخدَم من مسارات أخرى غير `/mushaf` كما هي)؛ البيانات الجديدة (مستوى الكلمة/السطر) تُضاف في مسار منفصل `public/data/quran-v2/` أو ما يعادله، لا تستبدل الملفات القديمة فعليًا.

## 7) مصدر البيانات الفعلي المُعتمَد لهذه الإعادة (بديل موثَّق لبوابة QUL المباشرة)

تحقّقتُ حيًّا (Playwright + curl) أن `qul.tarteel.ai` والمستودع الرسمي `TarteelAI/quranic-universal-library` على GitHub **لا يوفّران رابط تنزيل مباشر قابل للسحب آليًا** لخطوط QPC V2 لكل صفحة أو بيانات تخطيط الأسطر بلا تسجيل دخول (المستودع 698MB، تطبيق Rails كامل لا حزمة بيانات جاهزة).

**البديل المُتحقَّق فعليًا والمُعتمَد بدلًا منه:** واجهة `api.qurancdn.com` (API الإنتاجي العلني لـQuran.com، مبني على نفس بيانات QUL ونفس خطوط QPC V2 — يستخدمها موقع quran.com نفسه حيًّا، تحقّقتُ عبر مراقبة طلبات شبكة حقيقية لـ`quran.com/1`) توفّر:
- خطوط الصفحات: `https://quran.com/fonts/quran/hafs/v2/woff2/p{1..604}.woff2` (تحقَّقتُ من الصفحات 1، 2، 300، 603، 604 — كلها HTTP 200، `content-type: font/woff2`، ~40-190 كيلوبايت/صفحة).
- بيانات الكلمات/الأسطر: `https://api.qurancdn.com/api/qdc/verses/by_page/{n}?words=true&word_fields=text_uthmani,text_qpc_hafs&mushaf=2` — تُعيد لكل كلمة: `line_number` (1-15 الفعلي)، `text_qpc_hafs` (glyph مطابق لحرف الخط)، `text_uthmani`، بيانات الصفحة/الجزء/الحزب/السجدة كاملة.

هذا يُبقي الالتزام الكامل بروح البرومبت (بيانات QPC V2 الحقيقية من نفس منظومة QUL، لا توليد ولا تخمين) مع مسار تنزيل آلي فعلي بدل بوابة تسجيل دخول غير قابلة للأتمتة في هذه البيئة.
