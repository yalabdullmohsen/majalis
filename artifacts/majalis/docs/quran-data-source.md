# مصدر بيانات القرآن الكريم وسلامتها

## المصدر

نص المصحف في هذا التطبيق (رواية **حفص عن عاصم**، الرسم **العثماني**) مصدره
[AlQuran Cloud API](https://alquran.cloud/api) — واجهة برمجية مفتوحة ومجانية
بلا مفتاح API، إصدار `quran-uthmani`. نص AlQuran Cloud نفسه مبني على نص
[مشروع تنزيل (Tanzil.net)](https://tanzil.net/)، وهو النص المرجعي الأكثر
انتشارًا واعتمادًا في التطبيقات والمواقع الإسلامية، ويخضع لمراجعة دقيقة عبر
لجنة مختصة (راجع [Tanzil Quran Text](https://tanzil.net/docs/quran_text)
لتفاصيل منهجية المراجعة والتصحيح).

**لا يُعدَّل النص المصحفي في أي مرحلة من مراحل هذا المشروع** — لا عند الجلب
ولا عند التخزين ولا عند العرض. أي تطبيع (normalization) لأغراض البحث
(إزالة التشكيل، توحيد الألف، إلخ) يحدث في طبقة منفصلة تمامًا عند البحث فقط
ولا يُخزَّن ولا يُعرَض أبدًا كبديل عن النص الأصلي.

## البنية: محلي أولًا، ثم حيّ كاحتياط

حتى وقت قريب كان `src/lib/quran-api.ts` يعتمد حصريًا على استدعاء حيّ لكل
قراءة سورة. الآن:

1. **`scripts/fetch-quran-data.mjs`** يجلب الـ 114 سورة كاملة من AlQuran
   Cloud مرة واحدة (يدويًا، ليس ضمن `pnpm build`)، ويخزّن كل سورة حرفيًا
   كما وصلت من الـ API في `public/data/quran/surah-XXX.json`، مع بناء
   `public/data/quran/manifest.json` يحوي SHA-256 لكل ملف وإجمالي الآيات.
   يرفض السكربت الكتابة إن لم يكن العدد الإجمالي 114 سورة و6236 آية بالضبط.

2. **`fetchSurahDetail()`** في `src/lib/quran-api.ts` تحاول أولًا قراءة
   الملف المحلي المطابق (`/data/quran/surah-XXX.json`)؛ فإن نجحت القراءة
   استُخدمت مباشرة (أسرع، تعمل دون اتصال إن كانت الملفات مبنية ضمن الحزمة).
   إن فشلت القراءة (الملف غير موجود بعد، أو خطأ شبكة/تنسيق) ترجع الدالة
   تلقائيًا لطلب الـ API الحي تمامًا كما كانت تعمل من قبل — **لا يوجد أي
   احتمال لكسر الوظيفة** حتى قبل تشغيل سكربت الجلب أو في حال غياب سورة واحدة.

3. **`scripts/verify-quran-integrity.mjs`** يتحقق (بلا شبكة) من أن كل ملف
   محلي مطابق تمامًا لبصمة SHA-256 المسجَّلة في `manifest.json` وقت الجلب
   (يكشف أي تلاعب أو تلف لاحق)، ومطابقة عدد آيات كل سورة للعدد المرجعي في
   `SURAH_AYAH_COUNTS` (نفس المصدر المستخدَم في عرض قوائم السور)، ومطابقة
   الإجمالي الكلي 114 سورة / 6236 آية. مُضاف إلى `pnpm run test:regression`
   وإلى سلسلة `pnpm run verify:quran-data` المستقلة.

## كيفية تحديث البيانات المحلية

```bash
node scripts/fetch-quran-data.mjs      # يجلب الـ 114 سورة من AlQuran Cloud
npx tsx scripts/verify-quran-integrity.mjs   # يتحقق من السلامة بلا شبكة
```

الملفات الناتجة (`public/data/quran/*.json` و`manifest.json`) تُحفَظ في
المستودع كملفات ثابتة (Git) — لا تُولَّد تلقائيًا عند كل بناء، لتفادي
الاعتماد على شبكة/توفر الـ API وقت النشر، ولضمان أن البيانات المنشورة
مطابقة لما رُوجِع وتحقَّق منه.

## ما زال يعتمد على الجلب الحي (حدود معروفة)

الأجزاء التالية من الوحدة **لم تُنقَل بعد** لبيانات محلية، وتبقى تعتمد على
الطلب الحي من AlQuran Cloud وقت الاستخدام:

- **الترجمات والتفاسير** (`fetchTafsirAyahs`) — غير مربوطة أصلًا بأي واجهة
  عرض حاليًا (كود موجود بلا مستهلك)، ولو رُبطت مستقبلًا يلزم تسجيل مصدر كل
  تفسير وترخيصه وحالة المراجعة (`is_approved`) قبل عرضه.
- **القراءات العشر البديلة** (`fetchSurahDetailQiraat`) عدا حفص، ومعظمها
  أصلًا بلا `apiEdition` متاح في AlQuran Cloud (`null`) فتستخدم حفص احتياطًا.
- **الأجزاء (`fetchJuz`) والبحث في النص (`searchQuran`) وتحديد آيات كل
  صفحة (`fetchAyahsOnPage`)** — جميعها استدعاءات حيّة، لم تُنقَل محليًا في
  هذه المرحلة لأنها استعلامات مركّبة (جزء كامل / بحث نصي / صفحة) أعقد من
  مجرد "أعطني نص سورة واحدة"، وتحويلها يتطلب تصميم فهرسة محلية منفصلة.
- **صور صفحات المصحف** (KFGQPC عبر jsDelivr/GitHub) وملفات **الصوت**
  (everyayah.com، mp3quran.net) والبث المباشر — تبقى عبر CDN خارجي كما هي؛
  خارج نطاق "نص القرآن" وحجمها (آلاف الصور/ملفات الصوت) يجعل التخزين
  المحلي الكامل غير عملي حاليًا.

## لماذا لم نستبدل AlQuran Cloud كليًا بملف Tanzil الخام؟

الخيار المثالي طويل المدى هو تضمين نص Tanzil الخام مباشرة (بلا وسيط API)
مع رخصته الكاملة. اخترنا في هذه المرحلة تثبيت لقطة محلية من نفس المصدر
(AlQuran Cloud، القائم أصلًا على Tanzil) بدل استبدال المصدر بالكامل، للأسباب:

1. **تقليل المخاطرة**: AlQuran Cloud مُستخدَم في هذا المشروع منذ البداية
   ومُتحقَّق يدويًا (`scripts/verify-quran.mjs`)، فتثبيت لقطة منه بالتحقق
   الآلي (SHA-256 + عدد آيات) يحسّن السلامة والأداء دون تغيير مصدر الحقيقة.
2. **تفادي القرار الفقهي الصامت**: استبدال المصدر يتطلب مراجعة مستقلة
   للتأكد أن كل حرف مطابق لرواية حفص المعتمدة — وهذا قرار يحتاج مراجعة من
   مختص وليس تلقائيًا ضمن هذا التغيير.
3. **قابلية الرجوع**: يمكن لاحقًا استبدال `scripts/fetch-quran-data.mjs`
   بمصدر Tanzil مباشر دون تغيير أي كود استهلاك (`fetchSurahDetail` وشكل
   `SurahDetail` يبقيان كما هما) — التبديل معزول في نقطة واحدة.

إن رغب مالك المشروع باعتماد ملف Tanzil الخام مباشرة (مع نص الترخيص الكامل
المطلوب نشره)، هذا قرار منفصل يحتاج موافقة صريحة قبل التنفيذ.
